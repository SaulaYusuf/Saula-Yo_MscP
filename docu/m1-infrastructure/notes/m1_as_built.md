## 1. Initial Plan & Expectations

My original proposal called for a **multi‑channel Master‑Slave topology** – separate physical channels for logistics milestones (`masterchannel`) and high‑frequency IoT telemetry (`slavechannel`). I planned to use Hyperledger Fabric’s native **Channel** capability to isolate these data streams, expecting this to prevent consensus congestion on the global ledger.

I set up the local environment:
- **Ubuntu 22.04** (WSL2 on Windows)
- **Docker & Docker Compose**
- **Hyperledger Fabric v2.5.16** binaries and Docker images (I thought).

I followed the official `test-network` scripts, starting with the `./network.sh up -ca` command to generate crypto‑material using **Fabric Certificate Authorities (CA)**.

---

## 2. First Hurdle – Version Mismatch

When I ran the script, I saw:

```
LOCAL_VERSION=v2.5.16
DOCKER_IMAGE_VERSION=v3.1.5
Local fabric binaries and docker images are out of sync. This may cause problems.
```

The peer container (v3.1.5) was incompatible with my CLI tools (v2.5.16). I pulled the correct images manually:

```bash
docker pull hyperledger/fabric-peer:2.5.16
docker pull hyperledger/fabric-orderer:2.5.16
docker pull hyperledger/fabric-ca:1.5.17
```

I even tried tagging them as `latest` to force the script to use them, but the script still reported the mismatch because it checks the image version label, not the tag.

**Lesson:** The `network.sh` script does not honour environment variables like `IMAGE_TAG` consistently when using `-ca`. I later realised that using **cryptogen** (without `-ca`) is more stable and avoids the CA‑version complexity.

---

## 3. CouchDB Authentication Headache

When I used `-s couchdb` (to get rich queries for the Digital Twin), the peer container crashed with:

```
panic: Error in instantiating ledger provider: error handling CouchDB request. 
Error:unauthorized, Status Code:401, Reason:Name or password is incorrect.
```

I spent hours checking credentials, but the issue was that the peer’s `core.yaml` configuration (generated by the script) did not match the default CouchDB admin credentials. I eventually dropped CouchDB and reverted to **levelDB** – the simpler key‑value store – which is sufficient for the performance benchmarks I need. This decision removed the authentication error entirely.

---

## 4. The CA vs. Cryptogen Decision

After many failures with Fabric CA (including TLS CA path mismatches and missing `tlsca` directories), I decided to **switch to cryptogen** for crypto generation. This is the older, static method that the `network.sh` script handles reliably.

```bash
./network.sh down
docker volume prune -f
./network.sh up   # no -ca flag, uses cryptogen
```

This immediately stabilised the network. All peers and the orderer started without errors, and `peer channel list` showed the default `mychannel` was created and joined.

---

## 5. Custom Channel Nightmare

My goal was to create `masterchannel` and `slavechannel`. I ran:

```bash
./network.sh createChannel -c masterchannel
./network.sh createChannel -c slavechannel
```

The script generated the genesis blocks and reported “Channel created”, but the `peer channel join` step kept failing with:

```
Error: proposal failed (err: bad proposal response 500: cannot create ledger from genesis block: 
ledger [masterchannel] already exists with state [ACTIVE])
```

Yet `peer channel list` showed both channels were joined. The real problem emerged when I tried to deploy chaincode:

```
Error: failed to endorse proposal: rpc error: code = Unknown desc = 
error validating proposal: access denied: channel [slavechannel] creator org unknown, 
creator is malformed
```

The peer could not validate my identity against the channel’s MSP configuration. I realised the `network.sh` script is **heavily optimised** for the default channel `mychannel`. When you supply a custom channel name, the script’s internal environment variable scoping and MSP resolution break down – the channel’s genesis block does not correctly include the `Org1MSP` and `Org2MSP` definitions required for endorsement.

I tried manually copying TLS CA certificates and using `peer channel join` directly, but the same error persisted. After consulting the Hyperledger Fabric documentation and community forums, I confirmed that the `network.sh` script is not designed for robust multi‑channel provisioning; it’s a testing tool.

---

## 6. The Pivot Decision

**Time was running short.** I could have spent days writing custom bash scripts to manually create and join channels with correct MSPs, but that would have derailed my project timeline. My professor had warned: *“As you build, you will definitely change some things.”*

I made the **architectural pivot**:

- **Use a single channel** – `mychannel` – which the script handles perfectly.
- **Enforce logical segregation** via **different chaincode namespaces** on the same channel.
- **Route IoT data** to a chaincode called `slave-twin`.
- **Route logistics milestones** to a second chaincode called `master-logistics` (to be developed).

This still achieves the **Master‑Slave segregation** principle, but at the **application layer** rather than the network layer. The research gap – measuring whether isolating high‑frequency IoT telemetry from macro‑logistics prevents consensus bottlenecks – remains intact. In fact, by pushing both data streams through the *same* consensus pipeline, the performance metrics become more challenging, making a successful result even more compelling.

---

## 7. Stabilising with `mychannel`

I cleaned everything:

```bash
./network.sh down
docker volume prune -f
./network.sh up
./network.sh createChannel -c mychannel   # this succeeds fully
```

The peer joined successfully, and `peer channel list` confirmed `mychannel` was active. I now had a stable base network.
